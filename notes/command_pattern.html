Command design pattern

Design patterns describe simple and elegant solutions to common problems in object-oriented software design. One common design pattern is the Command pattern. It's mostly used when you want to execute a set of behaviors on a common object.

Few terms associated with the Command pattern that are used in this note:

  • The behaviors(commands) are encapsulated as objects
  • The object that initiates the execution of the commands is called the client
  • The object that the commands are acting on is called the receiver

Here are a few examples of how the command pattern could potentially be applied:

  • The relationship between a remote control & a TV. There are a set of commands (Turn on, off, increase volume and so forth) that are applied on the TV(the receiver) by the remote(client).
  • Apply a set of commands (save, edit, copy, paste) that are applied on some document(reciever) by pressing some icon(client).

Regardless of the situation, the goal is to create an object, the command object, that encapsulates the implementation of the desired behavior, and then have the user invoke the behavior only from the command object. This encapsulates the implementation of the behavior. The encapsulation is made possible by exposing just one method, usually named 'execute()'. The execution of the 'execute()' method subsequently causes some actions to be invoked on the receiver.

This behavior of the command pattern allows the requestor of the action to be decoupled from the object that carries out this action. It's very helpful in situations where there could be a number of ways to invoke an operation. For ex, you can execute any of the commands on a text document by either clicking a menu, an icon or even via a keyboard shortcut.

Another usecase of the command pattern is in situations when you might want to execute a set of commands synchronously on an object. If one command fails during this process, you can choose to immediately rollback the executed commands.

For example, if there is some microservice with several endpoints, and one of the endpoints will create some complex object 'A'. However, in order to create that object, the following things need to happen:

  1. the service has to call 5 other micro services to create parts of the object.(m1, m2, m3, m4, m5)
  2. There is a lot of dependency among the service calls. This means the call to service 2 depends on the response from service 1, for example.

So during the creation of the object, the command pattern can be used to encapsulate the the process of calling each micro service.

A set of commands can be defined, each responsible for sending a request to each microervice & processing the response. The order for the execution of each command needs to be set based on the dependency of each call. In this case, the reciever of the command can be an object that encapsulates the response from each call. Each command will have access to that object prior to the execution of the respective behavior. if any of the commands fail to execute, the commands that have already executed can be rolled back.

Commands can also be composed into Macro commands. This combines the Command pattern with the Composite pattern. More links to the composite pattern
